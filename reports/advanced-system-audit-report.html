## Executive Summary

The system is a mature TypeScript monorepo with strong baseline controls: Next.js route handlers, Prisma data model, shared entitlement logic, admin RBAC, CSRF/CSP/rate-limit middleware, connector outbox architecture, webhook validation surfaces, and broad Vitest coverage. The latest work added major Family/Pro workspace and connector capabilities, but those new surfaces are also where the highest risk sits.

No confirmed critical remote-code-execution style issue was found. The most important risks are:

- High: connector auto-dispatch can bypass the intended annual Pro entitlement.
- High: workspace sync can include a `fromAddressId` from outside the workspace or from soft-deleted history.
- High: `/api/profile` returns the raw selected `subscription` object, while that select includes Google Play `purchaseToken`.
- High: mobile purchase tokens are stored in plaintext even though the codebase already has field-level encryption patterns for connector tokens.
- Medium: Family/Pro billing and UI are only partially generalized; several settings components still hardcode Individual labels/prices, and scheduled plan changes do not persist target plan.
- Medium: Workspace seat limits, ownership transfer, destructive actions, and on-behalf sync need stronger atomicity and step-up controls.
- Medium: Android native config is stale versus Expo config, so invite app links and app-lock/permission readiness can drift in release builds.

## High Findings

### H-01: Auto Connector Dispatch Bypasses Pro/Annual Entitlement

Evidence:

- `apps/web/src/app/api/addresses/[id]/route.ts:101-102` enqueues connector sync when a primary address changes and connectors are globally enabled.
- `apps/web/src/lib/connector-runtime.ts:115-126` `enqueueAddressChange` loads addresses, user, granted consents, and connector configs but has no `userHasApiConnectorEntitlement` check.
- `apps/web/src/app/api/connector-dispatch/route.ts:21-25` shows the manual dispatch route does enforce `isApiConnectorsEnabled` and `userHasApiConnectorEntitlement`.

Impact:

A downgraded, expired, monthly Pro, Family, or Individual user who still has an existing granted consent can trigger partner updates through address edits even though the feature is meant to be annual Pro-only.

Exploit / failure path:

1. User previously grants connector consent while eligible.
2. User loses annual Pro entitlement.
3. User edits primary address.
4. Auto enqueue path still creates dispatch rows for granted consents.

Fix:

Centralize entitlement checks inside `enqueueAddressChange`, because every enqueue path flows through it. Route-level checks should remain as defense-in-depth.

Suggested patch shape:

```ts
if (!(await isApiConnectorsEnabled())) throw new Error("CONNECTORS_DISABLED");
if (!(await userHasApiConnectorEntitlement(input.userId))) {
  throw new Error("CONNECTOR_ENTITLEMENT_REQUIRED");
}
```

Add regression tests for auto-sync from address update, workspace sync, manual sync, downgraded Pro, monthly Pro, Family, Individual, and free trial.

### H-02: Workspace Sync Can Leak `fromAddressId` Across Workspace Boundaries

Evidence:

- `apps/web/src/app/api/workspaces/[id]/sync/route.ts:77-87` validates only `toAddressId` and only rejects mismatched non-null `workspaceId`.
- `apps/web/src/app/api/workspaces/[id]/sync/route.ts:90` passes `fromAddressId` to `enqueueAddressChange`.
- `apps/web/src/lib/connector-runtime.ts:121-124` loads `fromAddress` by `{ id, userId }` with no `deletedAt: null` and no `workspaceId` guard.
- `packages/db/prisma/schema.prisma:431-434` confirms `Address.workspaceId` is nullable during Workspace Phase 1.

Impact:

A permitted workspace sync can include origin-address PII from another workspace or from soft-deleted history in an external connector payload.

Exploit / failure path:

An owner/admin starts a workspace sync for a member and supplies a known `fromAddressId` belonging to that same user but outside the current workspace. The destination address passes validation, then runtime fetches the origin address without workspace isolation.

Fix:

Validate both `toAddressId` and `fromAddressId` against the same workspace and `deletedAt: null`. During the nullable migration phase, either block null `workspaceId` for workspace sync or route nulls through a controlled backfill-only path.

### H-03: `/api/profile` Exposes Raw Subscription Object Including Purchase Token

Evidence:

- `apps/web/src/lib/billing.ts:191-220` defines `SUBSCRIPTION_ENTITLEMENT_SELECT`; line `206` selects `purchaseToken: true`.
- `apps/web/src/app/api/profile/route.ts:85-99` returns `subscription` directly in JSON at line `93`.
- `packages/db/prisma/schema.prisma:247-248` stores both `purchaseToken` and `purchaseTokenHash`.

Impact:

An authenticated client can receive raw Google Play purchase tokens if present on the subscription row. Purchase tokens are provider credentials for subscription lookup/acknowledgement flows and should not be exposed to web/mobile clients.

Fix:

Return a sanitized subscription DTO from `/api/profile`. Keep `purchaseToken`, transaction identifiers not needed by UI, internal Stripe IDs, and schedule IDs server-only unless the UI has a specific safe need.

Suggested patch shape:

```ts
function publicSubscriptionDto(subscription: SubscriptionEntitlementRecord | null) {
  if (!subscription) return null;
  return {
    plan: subscription.plan,
    status: subscription.status,
    provider: subscription.provider,
    platform: subscription.platform,
    billingInterval: subscription.billingInterval,
    currentPeriodEndsAt: subscription.currentPeriodEndsAt,
    cancelAtPeriodEnd: subscription.cancelAtPeriodEnd,
    pendingBillingInterval: subscription.pendingBillingInterval,
    pendingBillingIntervalEffectiveAt: subscription.pendingBillingIntervalEffectiveAt,
  };
}
```

### H-04: Raw Mobile Purchase Tokens Are Stored Plaintext

Evidence:

- `packages/db/prisma/schema.prisma:247` stores `purchaseToken String? @db.Text`.
- `packages/db/prisma/schema.prisma:248` also stores a unique `purchaseTokenHash`.
- `packages/db/prisma/schema.prisma:1731-1732` shows the codebase already encrypts connector access/refresh tokens.
- `apps/web/src/lib/iap-common.ts` writes and reads raw `purchaseToken` in several IAP paths.

Impact:

Database read access exposes raw Google Play purchase tokens. This expands blast radius during database leaks, backup mishandling, admin debugging, or accidental JSON serialization.

Fix:

Prefer hash-only lookup plus encrypted raw token only where Google API refresh absolutely requires it. Rename to `purchaseTokenEncrypted`, encrypt with `FIELD_ENCRYPTION_KEY`, and keep `purchaseTokenHash` for lookup/dedupe. Migrate existing rows.

## Medium Findings

### M-01: Active Entitlement Can Stay Premium After Period End

Evidence:

- `packages/shared/src/entitlement.ts:322-359` computes `periodActive` for `ACTIVE`, but only denies access when `!periodActive && subscription.canceledAt`.
- If `status === "ACTIVE"`, `currentPeriodEndsAt` is expired, and `canceledAt` is null, line `343-359` still returns `hasAccess: true` and `hasPremium: true`.

Impact:

If a webhook/reconcile job misses a transition, a provider-paid subscription can remain premium indefinitely.

### M-05: Family/Pro Checkout Pending Rows Do Not Persist Plan/Provider/Access Type on Existing Rows

Evidence:

- `apps/web/src/app/api/stripe/checkout/route.ts:158-162` creates a new subscription with `{ userId, plan, status: "PENDING_CHECKOUT", provider: "STRIPE", platform: "web" }`.

A user upgrading from free/trial/manual states can temporarily hold a local row that says one plan/source while its checkout price indicates another. If the checkout is abandoned or webhook delayed, UI/support state can be misleading.

Store `pendingPlan`, `pendingProvider`, and `pendingPriceId`, or set a consistent pending state that the entitlement/UI layer understands.

### M-06: Past-Due/Grace Stripe Subscribers Can Start New Checkout

Past-due or grace-period Stripe users can be routed to create a new checkout instead of resolving the existing subscription in Stripe Portal. This can produce duplicate subscriptions or bypass dunning expectations.

Use the same blocking status set for Stripe-managed rows or send all rows with an existing `stripeSubscriptionId` to the portal/change-plan flow.

### M-11: Consent Revocation Races With In-Flight Dispatch

Evidence:

- `apps/web/src/lib/connector-runtime.ts:265-285` later writes worker result back unconditionally for the row.

Impact:

If a user revokes consent while a worker is calling a partner, the worker can still record `SUBMITTED` or `CONFIRMED` afterward.

### M-12: OAuth Token Exchange Bypasses Connector SSRF Controls

Evidence:

- `apps/web/src/lib/connector-oauth.ts:70-83` accepts runtime-config `authorizeUrl` and `tokenUrl`.
- `apps/web/src/lib/connector-oauth.ts:99-118` uses raw `fetch(config.tokenUrl)`.
- `packages/connectors/src/core/http-client.ts:87-100` has HTTPS and allowlist checks, but OAuth exchange does not use it.

Impact:

Runtime config compromise or misconfiguration can cause server-side requests to unintended hosts, including internal services, and lacks connector HTTP timeout/redirect hardening.

Fix:

Validate OAuth URLs against connector manifest allowlists. Enforce HTTPS, block private/link-local/loopback IPs after DNS resolution, set explicit timeout, and reject redirects outside the allowlist.

### M-13: Connector Allowlists Do Not Reject Private Hosts/IPs

Evidence:

- `packages/connectors/src/core/manifest.ts:37-40` only checks host lowercase/no slash/no spaces.
- `packages/connectors/src/core/http-client.ts:95-100` checks HTTPS and exact host match only.

Reject localhost, private IP literals, link-local, and internal domains at manifest validation. At request time, resolve hostnames and block private network destinations.

Evidence:

- `.env.example:80-83` includes Individual and Family mobile product examples.
- `.env.production.example:110-113` includes only Individual monthly/yearly.
- `packages/shared/src/runtime-config.ts:908-998` defines only Individual mobile product keys.
- `apps/mobile/app/settings/subscription.tsx:254-256` resolves only monthly/yearly SKUs for current purchase UI.

Impact:

Mobile store purchase surface is Individual-only while web pricing exposes Family/Pro. The app does link to web for Family/Pro, but runtime config and product docs are inconsistent.

Fix:

Either formally keep Family/Pro web-only and remove unused mobile Family examples, or add complete Family/Pro mobile product keys, product resolver logic, and store review tests.

## Low / Medium-Low Findings

### L-01: Admin Impersonation Redaction Comment Does Not Match Logic

Evidence:

- `apps/admin/src/lib/privacy.ts:181-184` comments "impersonation context is always SUPER_ADMIN-only" but redacts only when `!isAdmin`, meaning ADMIN can see `impersonatedByAdminId`.

Fix:

Redact unless `role === "SUPER_ADMIN"`.

### L-02: Workspace Invitation Duplicate Protection Is Race-Prone

Evidence:

- `packages/db/prisma/schema.prisma:1909` unique key is `[workspaceId, invitedEmail, expiresAt]`.
- `apps/web/src/app/api/workspaces/[id]/invitations/route.ts:96-103` does application-level pending-invite check before insert.

Impact:

Parallel invite requests with different expiration times can create duplicate pending invitations.

Fix:

Add a DB-level active/pending uniqueness strategy or transactional lock.

### L-03: Workspace Ownership Transfer Can Change Entitlement Anchor Abruptly

Evidence:

Workspace ownership and seat reconciliation use owner subscription as the entitlement source. Code comments and transfer helpers indicate billing anchor follows owner.

Impact:

Transferring a Family workspace to an Individual user can demote the workspace and put members into overflow/read-only state unexpectedly.

Fix:

Require transfer confirmation with plan impact, or require target owner to have sufficient plan before transfer.

### L-04: Removed Workspace Member Data Remains in Workspace by Design

Evidence:

`apps/web/src/app/api/workspaces/[id]/members/[memberId]/route.ts` deletes membership but comments that domain data and consent history remain.

Impact:

Owner/admin visibility into removed member data may be a privacy/product-policy issue unless explicitly disclosed.

Fix:

Define removal policy: retain, detach, anonymize, or require explicit export/delete consent. Add audit log and UI disclosure.

### L-05: Workspace Plan Feature Comment Mismatch

Evidence:

- `packages/shared/src/workspace-entitlements.ts:15-16` says manual connectors are "Family and up".
- `packages/shared/src/workspace-entitlements.ts:25-29` sets `INDIVIDUAL.manualConnectors: true`.

Impact:

Packaging and enforcement can drift.

Fix:

Update comment or plan matrix, and add a test asserting intended manual connector availability.

### L-06: Admin API Rate Limiter Is Per-Process

Evidence:

- `apps/admin/src/middleware.ts:378-416` uses an in-memory `Map`.

Impact:

In multi-replica production, limits are per instance and reset on deploy/restart.

Fix:

Use Redis/Upstash or another centralized limiter for admin API writes, login, cron, and internal endpoints.

### L-07: Some Mobile URLs Open Raw API/User-Provided Values

Evidence:

- `apps/mobile/app/custom-providers/[id].tsx` and `apps/mobile/app/services/[id].tsx` open website fields directly.
- `apps/mobile/app/providers/[id].tsx` has safer normalization for comparison.

Impact:

Malformed or non-web schemes can launch unexpected handlers or fail poorly.

Fix:

Centralize URL normalization and allow only `https:`/`http:` before `Linking.openURL`.

### L-08: Low-Confidence Dead Code / Transitional Code

Observed:

- `apps/web/src/lib/workspace-context.ts` states it is tested but unused/route retrofit comes later.
- Mobile unused-looking style/import entries: auth layout `useMemo`/theme imports, `oauthBtn` styles in sign-in/sign-up, delete-account unused styles.
- `packages/db/prisma/migrate-to-workspaces.smoke.ts` is not obviously wired into root/package scripts.
- Connector `FAILED` state exists in state tests, but persisted worker outcomes mostly flow through queued/dispatching/submitted/confirmed/needs-user.

Recommendation:

Use import graph tooling plus route/navigation-aware checks before deletion. Treat these as cleanup candidates, not immediate remove recommendations.

## Final Risk Statement

The current codebase is not "unsafe by default"; it has many good security and quality foundations. The risk is concentrated in newly expanded Family/Pro/workspace/connector paths where plan entitlement, workspace isolation, and billing state are not yet consistently enforced across every entry point. Fixing the centralized connector gate, subscription DTO/token storage, workspace sync validation, and Family/Pro billing state model should be treated as launch-blocking before broad Pro connector or Family workspace rollout.

# Current Product Logic Map

This document defines the current LocateFlow product contract for moving, providers, services, and move tasks. It is intentionally limited to the product that exists today plus approved current-product task tracking and user-created local providers.

It does not describe or approve Family, Pro, KYC, Plaid, USPS connectors, provider connectors, partner APIs, automatic external provider account updates, external address-change execution, or official provider partnerships.

## Provider

A provider is a listed service organization in the LocateFlow catalog. Providers can represent utilities, internet companies, banks, insurers, government agencies, local transit/toll organizations, subscriptions, retailers, and other move-relevant service categories.

Current provider records are directory listings unless source-backed validation is explicitly present. Showing a provider in LocateFlow does not mean:

- LocateFlow has a partnership with the provider.
- The provider is official or verified.
- The provider serves a user's exact address.
- LocateFlow can update the user's account with that provider.

Provider availability is guidance. Users must confirm details with the provider or an official source before acting.

## Custom Provider

A custom provider is a private user-created provider record for services that may not exist in the global provider catalog.

Examples include:

- Dentist.
- Law office.
- Physical therapy center.
- Local gym.
- Local utility.
- Local storage facility.
- Local daycare.
- Doctor or specialist.
- Pet veterinarian.
- Local parking provider.

Custom providers are private to the user by default. They are not global catalog records, verified provider records, official partner records, or source-reviewed provider records. Admin operators may review them for support and data governance, but promotion into the global catalog requires separate source review and product approval.

## Adding A Provider

Adding a provider means creating or linking a local LocateFlow service record so the user can track that service during a move.

Adding a provider does not:

- Connect to the provider account.
- Update the external provider account.
- Transfer service with the provider.
- Start or stop external service.
- Prove provider availability.
- Create an official relationship between LocateFlow and the provider.

The user experience must describe this as manual/local service tracking.

## Service

A service is a user's local LocateFlow record for a provider, custom provider, or provider name. It can be associated with an address and used in move planning.

Service state in LocateFlow is local application state. When a service is marked stopped, transferred, or address-updated inside LocateFlow, that only records what the user says happened or wants to track. It does not perform an external provider action.

## Moving Plan

A moving plan connects the user, origin address, destination address, and move dates. It is the current-product context used to decide which services may need move-related actions.

Changing a destination address can change:

- Provider coverage confidence.
- Suggested replacement providers.
- Move task recommendations.
- State/government guidance.
- Insurance and utility caveats.

It does not automatically update external providers.

## Move Task

A move task is a persistent LocateFlow task that helps the user track a move-related action.

Move task actions include:

- Stop old service.
- Start destination service.
- Transfer service.
- Update account address.
- Verify availability.
- Compare providers.
- Find replacement.
- Cancel or close.
- Government update.
- Insurance requote.
- Forward mail.
- No action.

Move tasks can be suggested by LocateFlow's deterministic classifier, created by the user, created by an admin, imported, or generated by system logic.

Move task status can be suggested, accepted, in progress, completed, dismissed, or reopened.

## Task Completion

Completing a task updates LocateFlow's local state where appropriate. Examples:

- A stop-service task may mark the local old service as inactive or stopped.
- A start-service task may create or link a destination service record.
- An update-address task may record that the user completed the address update.
- A verify-availability task may record that the user says they confirmed availability.

Task completion does not:

- Update an external provider account.
- Submit an address change to a provider.
- Start, stop, cancel, or transfer service outside LocateFlow.
- Prove official provider availability.

The UI must say that task completion updates LocateFlow only.

## Origin And Destination Addresses

Origin and destination addresses are used to:

- Compare same-state vs interstate moves.
- Infer move context.
- Match provider coverage by state and ZIP when available.
- Generate conservative task guidance.
- Display old-address and new-address task groupings.

ZIP and state are confidence signals, not proof of service availability.

## Provider Matching

Provider matching is deterministic. It is not ML-based.

The current matching system uses:

- Provider category.
- Provider scope.
- Provider active status.
- Provider state and ZIP data.
- Generated provider coverage rows.
- Static coverage metadata where present.
- Destination state and ZIP.
- Existing user services.
- State-rule context where available.
- Popularity and display order as secondary signals.

Provider matching must not present broad state or national coverage as address-level proof.

## ZIP, State, And Coverage Confidence

Coverage confidence is ordered conservatively:

1. Exact ZIP.
2. ZIP prefix.
3. Mapped service area.
4. State level.
5. National or federal.
6. Address check required.
7. Unknown.

Exact ZIP and ZIP prefix are still only catalog signals. Address-sensitive categories require caveats even when a ZIP signal exists.

Address-sensitive categories include utilities, internet/cable, trash/recycling, transit/toll, housing home services, local dining, local membership, and similar local services.

## Web Behavior

Web must:

- Show provider trust and coverage caveats.
- Say providers are listed/unverified unless source-backed validation exists.
- Say adding a provider is manual/local tracking.
- Show move tasks with status, confidence, caveats, and local-only completion language.
- Confirm before applying local task effects.
- Keep custom providers private/user-created in copy and UI.

## Mobile Behavior

Mobile must show the same product meaning as web:

- Same task statuses.
- Same action labels.
- Same provider trust labels.
- Same coverage confidence language.
- Same caveats.
- Same manual tracking disclaimer.
- Same no external automation disclaimer.

Mobile can use simpler layout, but it must not hide critical caveats.

Current mobile caveat: dedicated custom-provider list/detail/edit/delete screens exist, but the mobile launch verdict remains YELLOW until those routes are tested on real devices.

## Admin Behavior

Admin surfaces support current-product operations. Admin can inspect user services, custom providers, moving plans, move tasks, provider quality warnings, provider coverage gaps, and governance queues.

Admin actions must be permission-gated and audited where they change task/provider governance state.

Admin must not show providers as official or verified unless source-backed validation exists. Admin should treat user-created providers as private records unless explicitly reviewed.

Admin can view billing and entitlement context for support. Admin must not refund, cancel, grant grace, retry payment, grant/revoke entitlement, or perform other financial remediation unless a separate approved workflow exists.

## Billing Entitlement Defaults

Current-product billing defaults:

- Data export and account deletion are always available.
- Existing data remains readable after subscription expiration.
- Completing already-created move tasks remains allowed after subscription expiration.
- Generating new move tasks requires active entitlement.
- Creating new custom providers requires active entitlement.
- Admin billing surfaces are context-only unless a separately approved financial workflow exists.

These defaults preserve user access to their own data while keeping new paid workflow creation behind active entitlement.

## Email And Social Account Lifecycle

OAuth-only users can set a password through the supported account-security flow.

Safe email-change and social link/unlink workflows are not launch-ready. They must not be faked. Any UI that exposes email change or social link/unlink should be disabled or clearly marked unavailable until a safe flow exists with re-authentication, verification, audit logging, and session handling.

## Manual Work

The following remain manual:

- Provider account updates.
- Service transfers.
- Service starts/stops/cancellations.
- Provider availability checks.
- Official source validation.
- Logo/contact validation.
- Custom provider promotion into the global catalog.

## Official-Source Validation

Official-source validation is required before claiming provider data is verified, official, or source-backed. Accepted source types include official provider websites/contact pages, government/state agency pages, state utility commission data, FCC broadband data/maps, and licensed logo/brand sources.

Generic SEO pages, search snippets, unsourced directories, and inferred catalog data are not enough.

## Future Product Decisions

The following require separate product approval:

- Official provider verification workflow and schema.
- Provider partner status.
- External provider account linking.
- Automatic provider updates.
- USPS connector.
- Payment/provider integrations.
- Durable legal review workflow for state rules.
- ML ranking or learned provider recommendations.

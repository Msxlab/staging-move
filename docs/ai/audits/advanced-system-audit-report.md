## Verified Controls

### Authentication and Session Security

- User auth uses signed JWT plus DB-backed `UserLoginSession` token hashes and session activity checks in `apps/web/src/lib/user-auth.ts`.
- User JWT secret is required and length-validated in `apps/web/src/lib/user-jwt-secret.ts`.
- Web middleware verifies JWT at the edge, then route handlers re-check DB session state with `requireDbUserId` / `getUserSession`.
- Admin auth uses signed JWT plus DB-backed `AdminSession`, active admin checks, role checks, permission checks, password step-up, and MFA step-up in `apps/admin/src/lib/auth.ts`.
- Admin middleware enforces `ADMIN_JWT_SECRET`, IP rules, forced password change, MFA setup gate for privileged roles, session fingerprint checks, CSRF, CSP, and rate limits.
- Password login equalizes timing for unknown/OAuth-only users, rate-limits login attempts, supports MFA, and labels mobile sessions.
- OAuth callback handlers verify provider tokens, block unverified emails, block resurrection of soft-deleted users, and send linking/security notices.
- Mobile OAuth now requires PKCE at init and exchange in current code paths.

### Authorization and Data Ownership

- Workspace-scoped APIs use `resolveWorkspaceDataScope`, `requireWorkspaceContext`, `assertScopedRecordAction`, and shared permission logic to constrain member roles and shared data access.
- Address and service detail/update/delete routes check scoped ownership before returning or mutating records.
- Sensitive service fields are encrypted at rest, decrypted in route handlers, and redacted based on workspace role.
- Account deletion requires step-up, rate limits, audit/security events, optional grace-window restore, and calls a deletion orchestrator.
- Admin hard delete requires SUPER_ADMIN permission, password plus MFA step-up, target-bound email OTP, attempt limits, audit, and Stripe-cancel fail-closed behavior.

- Mobile IAP verification is server-side. iOS uses Apple signed transaction verification/refresh. Android verifies Google purchase token and product ID.

- EAS production profile enables iOS and Android store purchases, while development/preview profiles keep them off.

## Highest Verified Risks

### RISK-001: Production runtime startup still runs migrations

Severity: High operational risk.

Evidence:

- `package.json:8` defines `start` as `pnpm db:migrate:deploy && pnpm --filter @locateflow/web start`.
- `package.json:38` defines `db:migrate:deploy` as `prisma migrate deploy`.
- `Dockerfile:128` starts the runtime container with `prisma migrate deploy --schema packages/db/prisma/schema.prisma && exec node apps/web/server.js`.
- `.github/workflows/ci.yml:118-141` already defines a gated `Prisma Migrate (deploy)` job that runs after lint/typecheck, tests, and security.
- `docker-compose.prod.yml:54-84` and `docker-compose.prod.yml:151-152` already model a one-shot migrate service before app services.

Impact:

Runtime startup can perform production schema changes outside a dedicated deployment step. In multi-instance or auto-restart scenarios, app replicas may attempt migrations during service boot, increasing blast radius for bad migrations, lock contention, failed health checks, and drift from the CI-controlled migration boundary.

Recommendation:

Split production migration from app startup. Keep migration in CI/deploy one-shot jobs and make runtime `start` / Docker CMD only start the server after confirming schema readiness.

### RISK-002: Distributed rate limit, step-up, and lock safety depends on Upstash being configured

Severity: Medium operational/security risk.

Evidence:

- `apps/web/src/lib/rate-limit.ts:2-3` documents Upstash support with in-memory fallback.
- `apps/web/src/lib/rate-limit.ts:236-286` logs missing Redis in production and falls back to memory unless the caller passes strict fail-closed config.
- `apps/web/src/lib/rate-limit.ts:407-415` exposes `productionEnvOk` as false when production-like env lacks Redis.
- `apps/admin/src/middleware.ts:391-422` implements admin API rate limiting with a process-local `Map`.
- `apps/admin/src/lib/auth-step-up-store.ts` falls back to in-memory state when Redis is not configured, with a production warning.
- `apps/admin/src/lib/distributed-lock.ts` falls back to in-memory locks when Redis is not configured, with a production warning.

Impact:

In a multi-instance production deployment without Upstash, public/read/write limits, admin middleware route limits, step-up grace windows, and long-running admin-operation locks are process-local. Attackers or repeated clients can spread requests across instances, and operators can lose mutual exclusion guarantees for sensitive actions.

Recommendation:

Treat Upstash configuration as a production launch gate. Promote the health signal to deploy/readiness checks, and consider fail-closed or admin-readonly behavior for sensitive admin actions when distributed state is unavailable.

### RISK-003: SQL dump attachment filename uses database name without filename encoding

Severity: Low hardening risk.

Evidence:

- `apps/admin/src/app/api/backup/sql-dump/route.ts:420` builds `fileName` from `conn.database`.
- `apps/admin/src/app/api/backup/sql-dump/route.ts:426` interpolates that value directly into `Content-Disposition`.

Impact:

The route is SUPER_ADMIN-gated with password and MFA confirmation, and the database name comes from server config, not request input. Still, malformed or unusual database names could produce invalid response headers or confusing attachment names.

Recommendation:

Encode/sanitize the filename component and emit both `filename` and RFC 5987 `filename*`.

### RISK-004: Legacy partner consent refresh path is likely unreachable to cron callers

Severity: Low reliability risk.

Evidence:

- `apps/web/src/app/api/partner-consents/[id]/refresh/route.ts:8-14` defines a legacy cron-guarded path at `/api/partner-consents/[id]/refresh`.
- `apps/web/src/app/api/partner-consents/[id]/refresh/route.ts:10-11` says the canonical scheduler-safe route is `/api/cron/partner-consents/[id]/refresh`.
- `apps/web/src/middleware.ts:74-76` makes `/api/cron/` public to route-level cron auth, but does not public-allow `/api/partner-consents/` POST.

Impact:

Unauthenticated scheduler callers with `CRON_SECRET` should reach the canonical `/api/cron/...` route, but the legacy compatibility route sits behind session middleware before its own `guardCronRequest` can run.

Recommendation:

Either remove the legacy route, document it as session-protected/manual-only, or add an explicit middleware public exception if cron compatibility is still required.

### RISK-005: Mobile OAuth PKCE database column remains nullable after runtime enforcement

Severity: Low schema debt.

Evidence:

- `packages/db/prisma/migrations/20260512100000_mobile_oauth_pkce_challenge/migration.sql:19` adds `MobileOAuthCode.codeChallenge` as nullable.
- `apps/web/src/app/api/auth/oauth/google/route.ts:58-59` and `apps/web/src/app/api/auth/oauth/apple/route.ts:59-60` reject mobile OAuth init without a PKCE challenge.
- `apps/web/src/lib/mobile-oauth.ts:128-129` rejects exchange code creation without a valid challenge.
- `apps/web/src/lib/mobile-oauth.ts:200-207` rejects exchange of stored rows without a challenge or invalid verifier.

Impact:

Current runtime code enforces PKCE, so this is not an active bypass in inspected paths. The nullable schema leaves room for future/manual writes or legacy rows to exist and then fail at exchange.

Recommendation:

After confirming no legacy clients remain, backfill/purge old rows and migrate `codeChallenge` to `NOT NULL`.

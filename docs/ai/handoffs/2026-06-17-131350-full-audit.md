## Ranked Findings

### HIGH-1: Live Stripe configuration could not be verified from this audit context

Status: CONFIRMED limitation / launch blocker until manually verified.

Evidence:

- Local presence-only check found all requested Stripe env keys missing.
- Runtime config marks Stripe secret, webhook secret, publishable key, six Price IDs, and annual trial days as env-first billing keys at `packages/shared/src/runtime-config.ts:48-60`.
- Production readiness requires Stripe secret, webhook secret, publishable key, and all six price IDs at `apps/web/src/lib/production-readiness.ts:329-352`.
- Webhook handler returns 503 when `STRIPE_WEBHOOK_SECRET` is absent at `apps/web/src/app/api/webhooks/stripe/route.ts:587-590`.

Impact:

- I could not confirm the owner's statement that the live Stripe account and prices are connected.
- If any live price ID is wrong, inactive, wrong interval, wrong amount, or wrong currency, users can be charged a different amount than the app advertises.

Fix:

- Run the manual Stripe checklist above from an environment with read-only Stripe access or Dashboard access.
- Add an operator-only read-only billing health check that retrieves each configured Price and compares `unit_amount`, `currency`, `recurring.interval`, and `active` to `BILLING_PLAN_DEFINITIONS`.

### HIGH-2: Checkout does not verify live Stripe Price amount against advertised copy

Status: CONFIRMED.

Evidence:

- Family/Pro checkout resolves a configured Price ID at `apps/web/src/app/api/stripe/checkout/route.ts:177`, then passes it directly to Checkout at `apps/web/src/app/api/stripe/checkout/route.ts:240-249`.
- Individual checkout resolves a Price ID at `apps/web/src/app/api/stripe/checkout/route.ts:535`, parses display copy into local `firstChargeAmount` at `apps/web/src/app/api/stripe/checkout/route.ts:555-584`, then passes the Price ID directly at `apps/web/src/app/api/stripe/checkout/route.ts:626-636`.
- No `stripe.prices.retrieve` or amount/currency/interval comparison was found in `apps/web/src/app/api/stripe/checkout/route.ts`.
- Campaign validation does perform read-only Stripe Price retrieval and label comparison in admin code at `apps/admin/src/lib/stripe-campaign-validation.ts:100-165`.

Impact:

- A typo or stale Stripe Price ID can charge the wrong amount while the UI still advertises canonical labels.
- The app stores a copied display amount in `Subscription.firstChargeAmount`, but the actual charge is whatever the live Stripe Price object says.

Fix:

- Before creating Checkout sessions, retrieve the Price object read-only and verify active, fixed amount, USD, correct recurring interval, and expected unit amount for the selected canonical plan.
- Fail closed with a user-safe "plan temporarily unavailable" response if the Price object does not match.

### HIGH-3: App startup still runs database migrations

Status: CONFIRMED.

Evidence:

- Root start script runs `pnpm db:migrate:deploy` before web start at `package.json:8`.
- `db:migrate:deploy` invokes Prisma migrate deploy at `package.json:39`.
- Docker runtime `CMD` runs `prisma migrate deploy --schema packages/db/prisma/schema.prisma` before `node apps/web/server.js` at `Dockerfile:128`.

Impact:

- Scaling, restarts, and rollbacks can run schema migrations during app boot.
- A bad migration can block all app instances or run from multiple replicas.

Fix:

- Move migrations into a separate explicit deployment job/step.
- Start app containers only after migrations are verified complete.
- Keep app startup read-only against schema state.

### HIGH-4: Distributed rate-limit and admin step-up controls degrade to process-local memory when Upstash is absent

Status: CONFIRMED code behavior; production deployment state NOT VERIFIED.

Evidence:

- Web limiter is documented as falling back to memory when Upstash is not configured at `apps/web/src/lib/rate-limit.ts:1-4`.
- Missing Redis in production logs but uses in-memory mode unless a route is strict fail-closed: `apps/web/src/lib/rate-limit.ts:236-286`.
- Admin step-up store uses Upstash when configured and in-memory fallback otherwise at `apps/admin/src/lib/auth-step-up-store.ts:11-25`.
- The admin fallback warning says multi-instance step-up will not be coherent at `apps/admin/src/lib/auth-step-up-store.ts:74-85`.

Impact:

- In a multi-instance deployment without Upstash, rate limits, lockouts, and admin step-up confirmation windows are per-process.
- Attackers can bypass aggregate counters by hitting different replicas.

Fix:

- Treat Upstash Redis configuration as required for production/staging.
- Make production readiness fail if `UPSTASH_REDIS_REST_URL` / `_TOKEN` are absent in production-like deployments.
- Consider fail-closed for auth/admin groups when Redis is entirely unconfigured in production, after ops confirms availability expectations.
